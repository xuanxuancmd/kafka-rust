# 项目级规约

## 需求分析规约

+ 了解kafka的规格、代码范围、架构等任何问题都应该使用MCP工具deepwiki获得准确答案。

## 编码规约

1. 生成的代码文件和public函数命名需要和kafka 1:1对应，尤其是代码细节严格遵从kafka，禁止遗漏（比如可靠性细节）
2. 应该根据拆解的子任务，小步迭代，每个任务或子任务编码后都应该通过`cargo check`或`cargo build`命令成功后，再进行后续任务，编译和ut通过的结果应该展示给用户。**记住解决编译和测试问题时不应偷懒，而对已生成的代码大量删减**。
3. 代码生成，**禁止生成仅包含注释，但函数或代码段为空，或为默认值，或包含TODO的代码**。
4. 每个子任务编码且经过编译和测试运行后，还应该二次检查，确保代码生成无遗漏，无不满足第3条的情况。比如：你可以通过比较生成文件个数是否大致相同，以及代码行数是否偏差过大，从而进行核实。当然你可以通过其他方式二次检查。检查有问题应立即修复已完成的任务。
5. 如果编码过程中遇到无法自行解决的阻塞问题，你可以停顿并咨询我。
